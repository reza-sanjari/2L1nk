# Project Overview: Lightweight Self-Hosted Encrypted Communication System

## 1. Purpose and Core Goals

The project is a **self-hosted, single-binary communication system** that provides **secure real-time messaging and voice communication** through a web interface. The system prioritizes:

* Minimal resource usage
* Strong end-to-end encryption
* No required accounts or central identity provider
* Full control by the host
* Encrypted message persistence

The server acts strictly as a **transport and coordination layer** and is **never able to decrypt user content**.

---

## 2. High-Level Architecture

The system consists of three major components:

1. **Backend server**

    * Written in Go
    * Single statically compiled binary
    * Uses Echo for HTTP and WebSocket handling
    * SQLite database for persistence (always required — server cannot start without it)

2. **Client**

    * Pure JavaScript frontend
    * Runs entirely in the browser
    * Handles all cryptographic operations
    * Communicates via WebSockets and WebRTC

3. **Transport**

    * WebSockets for chat signaling and message delivery
    * WebRTC for peer-to-peer voice calls within rooms
    * Server used only for signaling and routing

---

## 3. User Identity Model

### Cryptographic Identity

* Each user is identified by a **public/private key pair**
* The public key fingerprint is the canonical user identity
* The private key never leaves the client

### Usernames

* Usernames are optional, user-chosen display labels
* Usernames are **not** used for authentication or encryption
* Usernames are associated with a public key
* Optionally signed by the user’s private key to prevent spoofing

---

## 4. User Modes

### Ephemeral (Guest) Mode

* Key pair generated in memory on page load
* Username chosen per session
* Keys are destroyed on refresh or tab close
* No persistence on the client
* The server does **not** save the user record to the database
* Messages the user sends are still stored in the database (under their fingerprint)

### Persistent User Mode

* Key pair generated once and stored in the browser
* Keys can optionally be encrypted with a passphrase
* User identity persists across reloads
* Username persists locally
* Server still does not store private keys

---

## 5. Server Access Control (Gate Model)

* On startup, the server generates a **gate key / access token**
* Access to the web interface requires knowledge of this token
* Prevents unauthorized users from connecting
* The gate key is not used for encryption, only access control
* Can be rotated by restarting the server

---

## 6. Communication Types

### 6.1 Rooms

All communication happens in rooms. A room can have any number of members — solo (1), duo (2), or multi-user. There is no separate DM concept; a 2-person conversation is simply a room with 2 members.

* Each room has a unique ID generated by the server
* Room membership is tracked by the server for routing only
* Encryption is handled entirely client-side

### 6.2 Voice Calls

* Voice calls happen within rooms via WebRTC
* Any members of the room can participate — not limited to 2 users
* Audio is peer-to-peer
* Server is only used for signaling (SDP/ICE exchange)
* End-to-end encrypted by WebRTC (DTLS-SRTP) natively

---

## 7. Cryptographic Design

### Key Types

1. **Identity Keys**

    * Long-lived Ed25519/X25519 key pair
    * Used for identity verification, gate authentication, and encrypting room keys to individual members
    * The private key never leaves the client

2. **Room Keys**

    * Random symmetric keys, one per room per epoch
    * Generated by the designated key creator and distributed to each member individually
    * Rotated on membership change (join or leave)

---

## 8. Encryption by Use Case

### Rooms (All Communication)

* One symmetric room key per room, per epoch
* Room key distributed to members by encrypting it with each member’s X25519 public key
* Messages encrypted using the room key (symmetric AEAD)
* Room key rotated on membership changes — epoch counter increments, key creator is reassigned
* Server only stores ciphertext and epoch number — it cannot decrypt any content

### Voice Calls

* WebRTC handles media encryption internally (DTLS-SRTP)
* Signaling messages may optionally be encrypted using the current room key

---

## 9. Message Transport and Routing

* Server assigns each client a connection ID
* Server maps:

    * Public key fingerprint → connection
* Messages contain:

    * Room ID
    * Encrypted payload
    * Sender identifier
    * Epoch number (which room key was used)
* Server routes messages based on IDs only
* Server never uses cryptographic material for routing

---

## 10. Persistence Model

The server always uses SQLite. There is no memory-only mode.

* Rooms and encrypted messages are always persisted to the database
* **Persistent users** (`UserModePersistent`): user record saved to the `users` table; identity survives reconnects
* **Ephemeral users** (`UserModeEphemeral`): user record is **never** written to the database; the server treats them as session-only
* No plaintext, no private keys, and no decryption keys are stored
* The server operator cannot read any stored messages

---

## 11. Privacy and Shutdown Behavior

* When the server stops:

    * Active WebSocket connections drop
    * In-memory hub state is lost (active sessions, room membership in memory)
    * The database (messages, rooms, persistent users) is intact and available on next start
* Ephemeral users are gone after disconnect — their record was never saved
* Clients do not retain messages unless explicitly designed to

---

## 12. Security Model and Threat Scope

### Protected Against

* Curious or compromised server
* Network eavesdropping
* Database leakage
* Unauthorized access without gate key

### Not Protected Against

* Compromised client devices
* Malicious browsers or extensions
* Nation-state adversaries
* Physical access to client devices

Security is based on:

* Strong, audited cryptographic primitives
* Minimal protocol complexity
* Clear separation between transport and crypto

---

## 13. Lightweight Design Constraints

* Single Go binary
* Minimal memory footprint
* No heavy dependencies
* Browser-native APIs
* Designed to run on low-power machines (e.g. Raspberry Pi)
* Protocol simple enough to be reimplemented on embedded clients

---

## 14. Extensibility Strategy

* Transport and routing APIs remain stable
* Cryptography evolves client-side
* Backend enforces structure, ordering, and membership
* Advanced features (ratcheting, key rotation, MLS-like behavior) can be layered on without redesigning the system

---

## 15. Project Scope Summary

This project intentionally focuses on:

* Self-hosting
* Explicit trust
* Ephemeral communication
* Minimal attack surface
* Understandable and auditable security

It does **not** aim to replace large-scale messaging platforms, but instead provides a **secure, private, lightweight communication tool** that can be fully controlled and understood by its users and operators.
